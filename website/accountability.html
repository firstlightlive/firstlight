ARCHITECTURE

WHY THIS CAN'T BE FAKED

8 enforcement layers. Server-side validation. GPS-verified proof. The admin would need to hack Strava's satellites to cheat.

01
🗑️ → ❌
DELETE BLOCKED
Database trigger raises an exception on ANY delete attempt. Not even the admin dashboard can remove a slip. The record is permanent.
02
🔒
FIELDS FROZEN
Date, rule, category, insight — all locked the moment a slip is created. The server rejects any attempt to modify core fields after insert.
03
📡
STRAVA GPS VERIFIED
The database reads distance directly from the Strava activity table — not from the browser. The admin can't type a fake number. The server pulls the truth from GPS.
04
🛰️
STRAVA DATA LOCKED
Only the automated Cloud Function can write Strava activities to the database. No human — not even the admin — can insert, edit, or delete a Strava record. The only source is Strava's API.
05
⏰
SERVER-SIDE ESCALATION
Escalation (+3 km/day after 7 days) is computed by the database using its own clock — not the browser. Changing your device time does nothing. The server decides the requirement.
06
📅
DATE ENFORCED
The Strava activity must have occurred on or after the slip date. You can't use an old walk from last month to clear a penalty logged today. The server checks the timestamp.
07
1️⃣
ONE ACTIVITY = ONE PENALTY
A unique database index physically prevents the same Strava activity from being used to clear more than one penalty. Each punishment requires its own dedicated walk.
08
⬆️
ONE-WAY STATUS
Penalty status can only move PENDING → CLEARED. Once cleared, it can never be reverted. Once proof is uploaded, it can never be changed. The state machine only moves forward.
